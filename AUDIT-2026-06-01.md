#### CRIT-1 ¬∑ `tools.py` ‚Äî `ALLOW_PATHS` does not restrict shell command file access (false security guarantee)

**Lines:** 1‚Äì15 (module docstring), `run_shell()` ~68‚Äì100

`ALLOW_PATHS` is advertised in the settings UI and security config as restricting what the shell tool can access. It only restricts the working directory (`cwd`). Any command that uses absolute paths bypasses it entirely:

```python
# ALLOW_PATHS = ["/home/user/.vespera/workspace"]
# This still succeeds:
run_shell("cat /etc/passwd")
run_shell("cp ~/.ssh/id_rsa /tmp/stolen.txt")
```

The `_path_allowed(resolved_wd)` check only validates `workdir`, not command arguments. The module docstring does acknowledge this ("File paths referenced inside the command itself are NOT path-checked"), but the settings UI description does not, and no warning appears in logs when a command with absolute paths runs. A user who sets `VESPERA_ALLOW_SHELL=true` and configures `VESPERA_ALLOW_PATHS` believing they are sandboxing the AI has no actual protection.

**Suggested fix:** Either:  
(a) Rename `VESPERA_ALLOW_PATHS` to `VESPERA_ALLOW_WORKDIRS` everywhere and update all UI descriptions to set correct expectations, OR  
(b) For `read_file` / `write_file` (which are already path-checked), keep the current behavior; for `shell`, add a prominent log warning each time the tool fires: "WARNING: shell=True grants unrestricted filesystem access ‚Äî ALLOW_PATHS does not restrict command arguments."  
Never present `ALLOW_PATHS` as a sandbox boundary for shell commands.

#### CRIT-2 ¬∑ `security.py` + `telegram_bot.py` ‚Äî Duplicate Telegram access control implementations that can diverge

**`security.py`:** `telegram_user_allowed()` function, lines ~55‚Äì59  
**`telegram_bot.py`:** `is_allowed()` function, lines ~39‚Äì44

There are two independent implementations of Telegram user access control. `telegram_bot.py` has its own `is_allowed()` that reads `TELEGRAM_ALLOWED_USERS` directly and does its own parsing. `security.telegram_user_allowed()` is **never called anywhere in the codebase** ‚Äî it is dead code.

If a developer modifies `security.telegram_user_allowed()` (e.g., adds logging, changes deny-by-default behavior, or adds a secondary check), those changes have zero effect because the bot doesn't call it. The bot's `is_allowed()` is the only thing that actually gates access.

Current behavior: both functions happen to be equivalent (both deny if empty list, both compare str(user_id)). Future maintenance could break this silently.

**Suggested fix:** Remove `is_allowed()` from `telegram_bot.py` and import `telegram_user_allowed` from `security.py`. Single source of truth.

#### CRIT-3 ¬∑ `api.py` ‚Äî `chat` endpoint stores sanitized message but passes original to `handle_message()`

**Lines:** ~222‚Äì226

```python
safe_message = _sanitize(message, 8000) or message
add_conversation(role="user", content=safe_message)   # sanitized version stored
result = handle_message(message)                       # ORIGINAL message to handler
```

The DB logs the sanitized version, but `handle_message()` receives the raw input. This is partially mitigated because `handle_message()` calls `_sanitize()` internally on the first line. However:

1. The `or message` fallback on line 1 is meaningless dead code ‚Äî `_sanitize()` never returns an empty string for non-empty input. It either returns the truncated text or the injection-removed placeholder. This pattern will mislead any future developer who reads it.
2. If `handle_message()` is ever refactored to remove its internal sanitization (which some developer might reasonably do since they see the caller "already sanitizes"), injection attempts would flow through undetected.
3. The conversation log and the actual AI processing input will be inconsistent any time `_sanitize()` transforms the message (injection detected, or truncation at 8000 chars produces different results for the log vs the handler).

**Suggested fix:** Pass `safe_message` to `handle_message()`, not `message`. Remove the dead `or message` fallback. Remove the redundant internal sanitization call inside `handle_message()` (or keep it as defense-in-depth, but document that clearly).

#### CRIT-4 ¬∑ `scheduler.py` ‚Äî `_callbacks` list modified and iterated from different threads without any lock

**Lines:** `register_callback()` ~140‚Äì142, `fire_reminder()` ~121‚Äì132

```python
_callbacks = []  # global

def register_callback(fn):
    _callbacks.append(fn)   # called from telegram_bot.py's main thread at startup

def fire_reminder(reminder):
    for cb in _callbacks:   # called from scheduler loop thread
        ...
```

`register_callback()` is called once at startup before the scheduler loop starts, so the race is usually benign. But it is a real race: if a callback is registered after the scheduler thread starts (e.g., a plugin or future feature), Python's GIL makes `list.append()` safe as an atomic operation, but **iterating a list while it's being appended to** can produce `RuntimeError: list changed size during iteration` in some Python versions and implementations.

**Suggested fix:** Use `threading.Lock()` around all `_callbacks` access, or replace the list with a `threading.local`-safe structure. At minimum, snapshot `list(_callbacks)` at the start of `fire_reminder()` to prevent mid-iteration mutation.

#### HIGH-1 ¬∑ `handoff.py` ‚Äî Module-level `CLOUD_API_KEY`, `CLOUD_MODEL`, `CLOUD_PROVIDER` are captured at import time and never refreshed

**Lines:** 15‚Äì21

```python
_cloud = COMPONENTS["cloud"]
CLOUD_PROVIDER = _cloud["provider"]
CLOUD_MODEL    = _cloud["model"]
CLOUD_API_KEY  = _cloud["api_key"]   # ‚Üê frozen at process start
```

`COMPONENTS` is populated from `os.getenv()` at `config.py` import time. `CLOUD_API_KEY` is a module-level string that is never updated. When a user updates the cloud API key via `/api/components/cloud` (which writes to `.env`), the change is explicitly documented to require restart. But the problem is **the user has no immediate feedback that nothing changed** ‚Äî they update the key, the API returns `{"ok": true}`, but the running system still uses the old key.

More practically: if `CLOUD_API_KEY` was empty at startup and the user adds it later via the UI, every call to `respond_cloud()` still returns the "no cloud API key ‚Äî falling back to local" path despite the UI showing the key is set.

**Suggested fix:** In `respond_cloud()`, read `get_component("cloud")["api_key"]` dynamically (or reload from `os.environ`) instead of using the module-level cached value. Since `api.py`'s `update_settings()` already calls `os.environ[key] = str(value)` for settings, a live re-read from `os.getenv()` inside `respond_cloud()` would pick up the new value without requiring restart.

#### HIGH-2 ¬∑ `handoff.py` ‚Äî Web search results not length-capped before prompt interpolation

**Lines:** ~230‚Äì236 (inside `handle_message()`)

```python
formatted_prompt = SEARCH_RESPONSE_PROMPT.format(today=today, results=results, message=message)
response = call_local(formatted_prompt)
```

`results` is the full return value of `web_search()`, which can be several KB (4 results √ó ~700 chars each = ~2,800 chars) plus the message and base prompt. Local Ollama models have a hard context limit (default 2048‚Äì4096 tokens for `llama3.2:3b`). When the prompt exceeds the context window, Ollama silently truncates input and/or produces degraded output.

Compare this to `background_loop.py` and `cleanup_crew.py` which explicitly pass `result[:500]` to avoid this. `handoff.py` does not.

**Suggested fix:** Truncate `results` before interpolation: `results=results[:2000]`. Also cap `message` to avoid double-overflow.

#### HIGH-3 ¬∑ `periodic_pruning.py` ‚Äî `_should_run()` / `_mark_ran()` not protected by any lock; double-run possible

**Lines:** `_should_run()` ~87‚Äì94, `_mark_ran()` ~96‚Äì100, `run_loop()` ~103‚Äì117

The background loop checks `_should_run()` at the top of each iteration, then calls `_mark_ran()` after running. The `/api/prune/run` endpoint also calls `run_pruning()` directly, bypassing this check entirely. If a manual API trigger fires while the background loop is mid-run, two pruning passes execute concurrently.

The `_pruning_lock` in `api.py` prevents duplicate API-triggered runs, but does nothing to prevent a concurrent background run. A manual API run and background run can overlap.

Additionally, `.pruning_last_run` is a plain text file on disk. `_should_run()` reads it and `_mark_ran()` writes it with no locking, no atomic write. If both processes read simultaneously before either writes, they both proceed. On a restart, the file may be partially written (file is not written atomically).

**Suggested fix:** Use an atomic write (write-then-rename pattern) in `_mark_ran()`, and add a module-level `threading.Lock()` to guard the check-run-mark sequence.

#### HIGH-4 ¬∑ `api.py` ‚Äî `.env` `$`-escaping via `_safe_env_value()` incompatible with python-dotenv

**Lines:** `_safe_env_value()` ~87‚Äì101, used in `update_component()` ~150‚Äì172 and `update_settings()` ~260‚Äì295

`_safe_env_value()` escapes `$` to `\$`, targeting bash semantics for double-quoted strings. But `python-dotenv` does **not** follow bash expansion rules. When the `.env` file is re-read by `dotenv.load_dotenv()`, a value like `sk-proj-abc\$123` will be returned as the literal string `sk-proj-abc\$123` (backslash preserved) rather than `sk-proj-abc$123`.

Test case: User has API key `sk-abc$def`. Write stores `"sk-abc\$def"`. On next process start, `load_dotenv()` returns `sk-abc\$def` ‚Äî wrong key, all cloud calls fail with 401.

**Suggested fix:** Use single-quoted `.env` values instead of double-quoted for all fields, or strip `$` escaping entirely and rely on single-quote quoting: `'{key}'\n` format, which dotenv handles correctly and bash interprets literally.

#### HIGH-5 ¬∑ `memory/store.py` ‚Äî Migration in `init_db()` runs every startup with no version tracking

**Lines:** `init_db()` ~45‚Äì60

```python
for col, typedef in [("used_cloud", "INTEGER DEFAULT 0"), ("complexity", "REAL DEFAULT 0.0")]:
    try:
        conn.execute(f"ALTER TABLE conversations ADD COLUMN {col} {typedef}")
    except sqlite3.OperationalError as exc:
        if "duplicate column name" not in str(exc).lower():
            raise
```

Every startup fires two `ALTER TABLE` statements and suppresses the "duplicate column name" error. This is functionally correct today, but it uses exception-swallowing as flow control, which is fragile. If SQLite changes its error message text (e.g., a locale change, or a different SQLite version), the `"duplicate column name"` string match fails, and the exception propagates and kills startup.

More critically: there is **no migration version table**. Adding future migrations will require expanding this pattern, and there's no way to know which migrations have already been applied other than trying each one and catching errors.

**Suggested fix:** Add a `schema_version` table. Apply migrations conditionally based on version number, not by catching duplicate-column errors. Use `CREATE TABLE IF NOT EXISTS` and `INSERT OR IGNORE INTO schema_version` patterns.

#### HIGH-6 ¬∑ `api.py` ‚Äî `TELEGRAM_ALLOWED_USERS` setting accepts arbitrary string with no validation

**Lines:** `_SETTINGS_SCHEMA` ~318‚Äì340, `update_settings()` ~360‚Äì410

`TELEGRAM_ALLOWED_USERS` is exposed as a free-form string setting via `/api/settings`. The schema `"type": "string"` means no format validation is applied. A value like `"abc, def, !!!"` would be accepted, written to `.env`, and would silently block all Telegram users on next restart (no numeric IDs match).

The setting also passes through `_safe_env_value()` which strips backticks and escapes `$` ‚Äî potentially corrupting a legitimately formatted ID string if it happens to contain those characters (unlikely for Telegram IDs, which are integers, but the system doesn't enforce this).

**Suggested fix:** Validate that `TELEGRAM_ALLOWED_USERS` values are comma-separated integers before writing to `.env`. Return a 400 error for invalid formats.

#### MED-4 ¬∑ `periodic_pruning.py` ‚Äî Background loop wait time does not adapt; restarts lose timing accuracy

**Lines:** `run_loop()` ~103‚Äì117

The loop calls `_shutdown.wait(interval)` at the end. If the process restarts 2 days after the last pruning run, `_should_run()` returns False (not 3 days yet), and the loop waits the full 3-day interval before checking again ‚Äî meaning pruning doesn't run for 5 days total. Depending on how frequently the process restarts (e.g., nightly system restarts), pruning could be perpetually deferred.

**Suggested fix:** After `_should_run()` returns False, calculate remaining time: `wait_time = interval - (time.time() - last_run)` and wait only that long.

#### MED-7 ¬∑ `utils.py` ‚Äî `_sanitize()` does not strip null bytes

#### MED-8 ¬∑ `handoff.py` ‚Äî Claude tool loop appends unboundedly to `messages` list; context overflow silently degrades output

**Lines:** `respond_cloud()` Claude section, ~135‚Äì195

Each tool call iteration appends the full assistant response + tool results to `messages`. With 10 max iterations and large tool outputs (e.g., `run_shell()` returning up to 32,000 chars), the messages list can easily exceed Claude's context window. This produces a `400 / context_length_exceeded` API error which is caught and returned as the generic error string ‚Äî providing no diagnostic information.

**Suggested fix:** Track cumulative message token count (rough estimate: `sum(len(str(m)) for m in messages) // 4`), abort the tool loop early with a user-facing explanation if approaching the limit (e.g., 80,000 chars estimated tokens), and log the actual error message rather than masking it.

#### MED-9 ¬∑ `web_search.py` ‚Äî `_is_price_query()` has false positive patterns for "eth" and "oil"

**Lines:** `_PRICE_PATTERNS` dict, ~37‚Äì55

`re.compile(r"\beth\b")` matches standalone "eth" which appears in legitimate non-cryptocurrency contexts ("Give me eth... a moment"). `re.compile(r"\boil\b")` similarly matches "oil paint", "baby oil", etc. When a false positive fires, `_fetch_price("ETH-USD")` or `_fetch_price("CL=F")` is called, adding latency and returning a confusing price result.

The `_PRICE_PRE_CHECK` regex in `handoff.py` is more conservative (requires price keyword adjacent to asset name), but `web_search.py`'s `_is_price_query()` only requires the asset word anywhere in the query plus any price keyword anywhere in the query.

**Suggested fix:** Require asset word and price keyword within N words of each other, or use the same stricter regex pattern already present in `handoff.py`.

#### LOW-1 ¬∑ `api.py` ‚Äî `safe_message = _sanitize(message, 8000) or message` ‚Äî dead code fallback

**Line:** ~222

The `or message` fallback fires only if `_sanitize()` returns an empty string. For non-empty input, `_sanitize()` always returns either the truncated text (non-empty, since `len(message) >= 1`) or the injection-removed placeholder string (non-empty). Since `message` has already been validated as non-empty and ‚â§8000 chars, this fallback is unreachable. #### LOW-2 ¬∑ `background_loop.py` / `cleanup_crew.py` / `periodic_pruning.py` ‚Äî `global _shutdown` reassignment pattern is fragile

**`background_loop.py` line ~83**, similar in others:

```python
_shutdown = threading.Event()

def run_loop(shutdown_event: threading.Event = None):
    global _shutdown
    if shutdown_event:
        _shutdown = shutdown_event
```

Reassigning the module-level `_shutdown` via `global` means the original `_shutdown` object (which may be referenced by other code) is no longer the active one. This pattern works as long as no external code holds a reference to the original event. Fine now, fragile as the codebase grows.

**Suggested fix:** Accept `shutdown_event` as a required parameter and don't maintain a module-level default. Or use `if shutdown_event is not None: shutdown_event.wait(interval)`.

- **`api.py`** ‚Äî Rate limiter design is solid: per-IP deque with sliding window, LRU eviction cap at 10,000 IPs, separate `_rate_lock`, `Retry-After` header on 429. `_get_client_ip()` correctly handles `X-Forwarded-For` with opt-in `TRUST_PROXY`. **Overall assessment:**
